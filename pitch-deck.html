02 / 22 PARTE 1 · Cenário Atual

Quatro sócios,
quatro realidades distintas

Cada operação cresceu do seu jeito — com seu porte, suas ferramentas e sua maneira de controlar o dia a dia.

Estágios diferentes

De 27 a 120+ UCs, faixas de potência distintas, maturidade operacional diferente. Não existe um único "padrão" entre os sócios.

Ferramentas próprias

Planilhas, ERPs leves, controles próprios. Cada um encontrou um jeito que funciona — para o tamanho atual da sua operação.

Trajetórias independentes

Cada sócio define seu próprio ritmo de captação, sua carteira e sua estratégia comercial. Os caminhos não se sobrepõem.

O ponto de partida não é uma "dor comum" — é a constatação de que cada sócio tem o seu cenário, e qualquer solução precisa respeitar isso.
03 / 22 PARTE 1 · Cenário Atual

O que ainda não existe
é a camada comum

Cada sócio tem o controle da sua própria operação. O que falta é uma base compartilhada — quando, e se, fizer sentido.

Visão individual sólida

Cada sócio sabe o que acontece dentro da sua carteira. A operação roda. O que muda com uma plataforma é o nível de profissionalização e a capacidade de escalar sem refazer tudo.

Mesma plataforma, tenants segregados

Cada sócio opera dentro do seu próprio tenant — dados, clientes e processos totalmente isolados —, mas todos acessando a mesma plataforma. Autonomia preservada, sem mistura de operações.

A plataforma não substitui o que cada um já faz bem. Ela cria o terreno comum para o que vier depois.
04 / 22 PARTE 1 · Cenário Atual

O ponto em comum:
o próximo salto de escala

Operações diferentes esbarram, mais cedo ou mais tarde, nos mesmos limites estruturais quando crescem.

Escala

Dobrar de tamanho costuma exigir dobrar a equipe administrativa — a menos que processo e ferramenta evoluam junto.

Padronização

Cada novo cliente reaproveita pouco do anterior. Padronizar reduz retrabalho e libera tempo para vender e atender.

Continuidade

Backup, controle de acesso, histórico auditável e LGPD passam a importar à medida que a base de clientes cresce.

Não é uma urgência para hoje — é a infraestrutura para o próximo ciclo de crescimento de cada sócio.

09 / 22 PARTE 3 · Pilar 1

Multitenancy com
Isolamento Total

O que isso significa tecnicamente

  • Cada sócio tem seu próprio banco de dados no SQL Server (multibanco)
  • Não é "filtro de tenant_id" — é separação física
  • Identidade isolada via Keycloak
  • Backup e restore independentes
  • Cada sócio pode ter seu próprio domínio

O que isso significa pra vocês

  • ✓ Dados de um sócio nunca vazam pro outro
  • ✓ LGPD-compliant por arquitetura
  • ✓ Performance isolada (um não trava o outro)
  • ✓ Sócio pode sair levando o banco dele
  • ✓ Auditoria separada e rastreável
Isolamento de dados não é configuração, é arquitetura. Cláusula de segurança jurídica embutida.

12 / 22 PARTE 4 · Mitigação de Risco

Arquitetura DDD + CQRS
em Monolito Modular

Decisões maduras pra reduzir custo de operação e risco de manutenção.

Monolito Modular

Deploy simples, infra barata, evolução rápida. Sem complexidade desnecessária de microserviços hoje — mas pronto pra separar quando justificar.

DDD

O código fala a linguagem de vocês: "Crédito de Energia", "Rateio", "Beneficiário". Manutenção barata e onboarding rápido.

CQRS

Leitura separada de escrita. Dashboards rápidos mesmo com milhares de UCs. Pronto pra escalar sem refatoração.

Stack mainstream (.NET 8 + Vue 3 + SQL Server) — qualquer fornecedor sério dá continuidade.
13 / 22 PARTE 4 · Garantias

Os 4 medos — e como protegemos vocês

"E se atrasar?"

Pagamento 30/30/40 atrelado a entregas, não a calendário. Vocês não pagam o próximo bloco até receber o anterior em mãos.

"E se a equipe sumir?"

Código entregue a cada sprint no repositório de vocês. Documentação completa + treinamento da equipe inclusos.

"E se a tech ficar obsoleta?"

Stack mainstream (.NET 8, Vue 3, SQL Server) — qualquer fornecedor pega a continuidade. Sem lock-in técnico.

"E se um sócio sair?"

Banco de dados dedicado: exporta o banco inteiro dele e entrega. Sem trauma técnico, sem trauma jurídico.

Cada medo tem uma cláusula contratual ou técnica. Os riscos foram pensados antes — não depois do problema.
